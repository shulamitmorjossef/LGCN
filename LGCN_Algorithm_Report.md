# דוח טכני: אלגוריתם LGCN לאופטימיזציית מסלולי נהיגה

## תקציר

הפרויקט מציע פתרון לבעיית **תכנון מסלולים דינמי לנהג בודד** (Single-Vehicle Dynamic Routing Problem) בעלת אילוצי חלון-זמן (Time Windows), עומסי תנועה משתנים (Traffic) ועדיפויות-תחנה (Priorities). הפתרון מבוסס על מודל ניורוני בשם **LGCN — Location-based Graph Capsule Network**, המשלב רעיונות מתחום **רשתות קפסולות (Capsule Networks)** ו-**ניתוב דינמי בהסכמה (Dynamic Routing by Agreement)** עם בנייה גרידית (Greedy) של המסלול הסופי. המודל פרוס מאחורי שרת **FastAPI**, ושומר עבור כל נהג גרף-מצב דינמי (Dynamic Graph) הנשמר לדיסק ומתעדכן באופן אינקרמנטלי בין קריאות.

---

## 1. הגדרת הבעיה

המערכת פותרת גרסה מצומצמת ויישומית של בעיית **VRPTW** (Vehicle Routing Problem with Time Windows) עבור נהג בודד:

- לנהג יש **מיקום נוכחי** (Current Location) ורשימת **תחנות** (Stops) שעליו לעבור בהן.
- לכל תחנה יש קואורדינטות (`x`, `y`), **חלון זמן** (`tw_start`, `tw_end`) ו**זמן המתנה** משוער (`wait`).
- בין כל שתי נקודות יכול להיות **עיכוב תנועה** (`traffic_delays`) ול**תחנה** יכולה להיות **עדיפות** (`priority`).
- המטרה: לקבוע **סדר ביקור אופטימלי** בתחנות, כך שתחנות דחופות (חלון זמן צר, דדליין מוקדם, המתנה קצרה, עדיפות גבוהה) ייבקרו מוקדם יחסית, מתוך איזון עם מרחק/עומס תנועה.
- הבעיה היא **דינמית**: בכל קריאה חדשה יכולות להתווסף תחנות, מיקום הנהג מתעדכן (ומכיל "רעש GPS"), ויש להבחין בין תחנה חדשה לבין תחנה קיימת שכבר טופלה.

## 3. רקע תיאורטי: Capsule Networks ו-Dynamic Routing

המודל שאול מ"רשתות קפסולות" (Sabour et al., 2017), שתוכננו במקור לסיווג תמונות, ומותאם כאן לבעיית ניתוב.

### 3.1 קפסולה (Capsule)
קפסולה היא וקטור (ולא סקלר בודד כמו ב-CNN רגיל) המייצג ישות. **הכיוון** של הוקטור מקודד תכונות (geo, זמן, דחיפות...), ו**האורך** שלו (לאחר squash) מקודד "הסתברות"/חוזק קיום.

### 3.2 פונקציית Squash
ממומשת ב-[capsule.py](lgcn/capsule.py):

```python
def squash(v):
    norm = torch.norm(v, dim=-1, keepdim=True)
    return (norm**2 / (1 + norm**2)) * (v / (norm + 1e-8))
```

הפונקציה "מכווצת" את אורך הוקטור לטווח `[0,1)` בלי לשנות את כיוונו — וקטורים קצרים נכווצים לכמעט אפס, וקטורים ארוכים מתקרבים ל-1. כך אפשר לפרש את אורך קפסולה כמדד עוצמה/ביטחון.

### 3.3 Dynamic Routing by Agreement
ממומש ב-[routing.py](lgcn/routing.py):

```python
def dynamic_routing(u_hat, iters=3):
    b = torch.zeros(u_hat.size(0), u_hat.size(1))
    for _ in range(iters):
        c = torch.softmax(b, dim=1)
        s = (c.unsqueeze(-1) * u_hat).sum(dim=0)
        v = squash(s)
        b = b + (u_hat * v).sum(dim=-1)
    return v
```

האלגוריתם (3 איטרציות):
1. `u_hat` — "חיזויים" (votes) של כל קפסולת-קלט לקפסולת-הפלט.
2. `c = softmax(b)` — משקלי-צימוד (coupling coefficients) המתחילים אחידים.
3. `s` — סכום משוקלל של כל החיזויים → קפסולת-הפלט הגולמית.
4. `v = squash(s)` — נרמול הקפסולה.
5. `b` מתעדכן לפי **הסכמה** (dot-product) בין כל חיזוי `u_hat` לתוצאה `v`: חיזויים שה"הצביעו" בכיוון הדומה לתוצאה מקבלים משקל גבוה יותר בסבב הבא.

המשמעות התפעולית: צמתים (תחנות) ש"מצביעים" בכיוון דומה (geo קרוב, חלון-זמן דומה, דחיפות דומה) **יתכנסו** לאותה קפסולת-פלט (subroute), בדומה לאשכול (clustering) רך מבוסס-הסכמה.

## 4. ארכיטקטורת מודל ה-LGCN

מוגדר ב-[lgcnModel.py](lgcn/lgcnModel.py). שכבות המודל, מהקלט לפלט:

```
node_features (N×5)
      │  Linear  W1 [5→8]
      ▼
  node_caps (N×8)  ──squash──▶ קפסולת-צומת
      │  RoutingLayer (Linear W2 [8→4] + dynamic_routing)
      ▼
 subroute_caps (1×4)           קפסולת תת-מסלול (עם num_subroutes=1: צובר/אשכול יחיד)
      │  RoutingLayer (Linear W3 [4→4] + dynamic_routing, num_subroutes=1)
      ▼
 full_route_caps (1×4)         קפסולת המסלול השלם — "טביעת אצבע" של כל הבעיה
      │
      │  (במקביל, מקפסולת-הצומת:)  Linear  W4 [8→4]  →  u_hat (N×4)
      ▼
 sim[j] = dot(u_hat[j], full_route_caps)        ◀── ציון "שייכות"/דחיפות לכל תחנה j
```

### 4.1 קלט: `node_features` — 5 תכונות לכל תחנה
`[x, y, tw_start, tw_end, wait]` — קואורדינטות, חלון זמן, וזמן המתנה משוער.

### 4.2 שכבה 1 — `node_fc` (W1, גודל 8×5)
ממפה כל תחנה לקפסולת-צומת בגודל 8. כל שורה במטריצה (ראו תיעוד מפורט ב-[main.py:12-37](main.py#L12-L37)) מקודדת תכונה נגזרת:

| מימד | מה הוא מקודד |
|---|---|
| 0–1 | מיקום גיאוגרפי מנורמל |
| 2 | מרכז חלון הזמן |
| 3 | רוחב חלון הזמן (דחוף → קרוב ל-0, גמיש → שלילי-גדול) |
| 4 | דדליין (`tw_end`) — מוקדם יותר → ערך גבוה יותר |
| 5 | תחילת חלון (`tw_start`) |
| 6 | זמן המתנה (פחות המתנה → ערך גבוה יותר) |
| 7 | ניקוד דחיפות משולב (geo + חלון + דדליין + wait) |

זוהי שכבת **Feature Engineering נלמדת** (כאן עם משקלים קבועים, ראו סעיף 6) שמתרגמת תכונות גולמיות לייצוג שמבטא "דחיפות".

### 4.3 שכבה 2 — `RoutingLayer` (W2, גודל 4×8) + Dynamic Routing
מטילה את קפסולת-הצומת (8) לקפסולת-הצבעה (4), ולאחר מכן מריצה `dynamic_routing` על כל הצמתים יחד. כל שורה ב-W2 מגדירה "אשכול" סמנטי: גיאוגרפי, זמני, דחיפות, ויעילות-כוללת (ראו [main.py:39-53](main.py#L39-L53)). תחנות עם תכונות דומות "מצביעות" באופן דומה ולכן מתכנסות לאותה קפסולת תת-מסלול.

### 4.4 שכבה 3 — `full_route_routing` (W3, גודל 4×4)
ממזגת קלות בין מימדים (geo↔זמן, דחיפות↔יעילות) ומפיקה קפסולת **מסלול שלם** אחת — מעין תקציר וקטורי של כל הבעיה.

### 4.5 הטלת דמיון — `proj` (W4, גודל 4×8)
מטילה כל קפסולת-צומת (8) למרחב של `full_route_caps` (4), ומחשבת ציון `sim[j] = dot(u_hat[j], full_route_caps)`. ה-Sign Convention שנבחר (ראו [main.py:67-86](main.py#L67-L86)) גורם לכך ש**תחנה דחופה יותר מקבלת ציון `sim` גבוה יותר** — כיוון שערכי מימדי-הדחיפות בקפסולה שליליים, ומכפלת-נקודה של שני וקטורים שליליים-באופן-דומה גדולה יותר עבור תחנות דחופות.

> כל ארבע המטריצות (`W1`–`W4`) **קבועות מראש** (Hand-crafted) ולא מאומנות מ-Backpropagation בפועל בקוד הנוכחי — הן ממומשות כמשקלי `nn.Linear` קפואים (`lgcn.eval()`), כך שהמודל הוא בפועל **היוריסטיקה דטרמיניסטית עם מבנה רשת קפסולות**, שמיועדת לאפשר בעתיד אימון (Backprop על דוגמאות אמיתיות) מבלי לשנות את הארכיטקטורה.

## 5. בניית המסלול הסופי — Greedy Construction

לאחר קבלת `sim[j]` לכל תחנה, נבנה המסלול באופן **גרידי** ([main.py:342-360](main.py#L342-L360)):

```python
for _ in range(n - 1):
    last = route[-1]
    best = argmax_j( sim[j] - dist[last,j] * (delay[last,j] + priority[j]) )
    route.append(best)
```

כלומר, בכל צעד, התחנה הבאה היא זו שממקסמת:

```
score(j) = sim[j]  −  dist(last, j) × ( traffic_delay(last, j) + priority(j) )
```

- **`sim[j]`** — "פרס" המתקבל מהמודל (דחיפות/חשיבות תחנה j).
- **`dist(last, j)`** — מרחק אוקלידי מהתחנה האחרונה בנתיב.
- **`traffic_delay`** — עיכוב תנועה ספציפי (אם סופק).
- **`priority(j)`** — עדיפות חיצונית (גבוה יותר ⇒ "עולה" יותר, ולכן מרחיק את j מהיעד הבא, בדומה לעונש).

זהו אלגוריתם **Greedy Nearest-Insertion מורחב**, המשקלל מודעות-מודל (capsule similarity) עם עלות מעבר בפועל (מרחק × (תנועה+עדיפות)). מורכבות זמן ריצה: `O(n²)` (n תחנות), המתאים לכמות תחנות קטנה-בינונית הטיפוסית לנהג בודד.

## 6. ניהול מצב דינמי — `DynamicGraph` ו-Driver State

### 6.1 מבני נתונים
- **`Node`** ([graph/node.py](graph/node.py)) — תחנה: מיקום, ביקוש, חלונות-זמן, זמן המתנה.
- **`Edge`** ([graph/edge.py](graph/edge.py)) — קשת בין שתי תחנות: מרחק, תנועה, דחיפות. `weight() = dist × (traffic + urgency)`.
- **`DynamicGraph`** ([graph/dynamicGraph.py](graph/dynamicGraph.py)) — אוסף צמתים/קשתות; בהוספת צומת חדש נבנות **אוטומטית** קשתות לכל הצמתים הקיימים (גרף מלא, Complete Graph).

### 6.2 שמירת מצב לכל נהג (Caching)
מטופל ב-[main.py:118-180](main.py#L118-L180): כל נהג מקבל קובץ JSON עצמאי (`driver_states/<driver_id>.json`) עם רשימת התחנות שכבר טופלו. כך הגרף **לא נבנה מאפס** בכל קריאה — תחנות שכבר נראו מזוהות לפי קואורדינטות (`_find_matching_node`, סובלנות `1e-6`) ולא נכפלות.

### 6.3 מיקום נוכחי (Current Location) — לוגיקה מיוחדת
`new_points[0]` הוא תמיד מיקום הנהג העכשווי, ומקבל **מזהה sentinel קבוע `-1`** ([main.py:215-217](main.py#L215-L217)). הוא:
- **לא** נשמר ב-cache (כדי שרעש GPS בין קריאות לא ייצור תחנות-רפאים).
- משמש כנקודת **התחלה** של חישוב המסלול בכל קריאה.

זו נקודת תכנון חשובה: היא מאפשרת **עדכון אינקרמנטלי** — לקוח יכול לשלוח בכל פעם את כל התחנות הנוכחיות (כולל ישנות), והשרת מסנן את הכפילויות לבד, בלי שהלקוח יצטרך לדעת מה כבר טופל בעבר.

### 6.4 זרימת `optimize_route` (סיכום שלבים)
1. טען גרף-מצב קיים לנהג (או גרף ריק אם נהג חדש).
2. עבור כל תחנה נכנסת — אם קיימת (קואורדינטות תואמות) שייך אליה; אחרת — צור צומת חדש והוסף לגרף, ושמור.
3. בנה את "כל הצמתים" = מיקום נוכחי (transient) + כל התחנות בגרף.
4. הפק תכונות לכל הצמתים והרץ אותן במודל ה-LGCN (Inference, ללא Gradient — `torch.no_grad()`).
5. חשב מטריצת מרחקים, ומפה `traffic_delays`/`priorities` (אם נשלחו) למיקומים בגרף.
6. הרץ בניית מסלול גרידית כמתואר בסעיף 5.
7. החזר רשימת מזהי-צמתים בסדר המסלול (`-1` = המיקום הנוכחי).

## 8. Baseline להשוואה

[baseLine/greedy.py](baseLine/greedy.py) ממש **Nearest-Neighbor Greedy** קלאסי — בכל שלב בוחר את התחנה הקרובה ביותר גיאוגרפית, בלי להתחשב בחלון-זמן, דחיפות או תנועה. הוא משמש קו-בסיס (baseline) להמחשת התרומה של LGCN: ב-Nearest-Neighbor תחנה עם דדליין קרוב אך מרוחקת עלולה "להישאר אחרונה" ולפספס את חלון הזמן שלה, בעוד LGCN משקלל זאת באמצעות `sim[j]`.

## 9. נקודות חשובות לדגש בספר הפרויקט

1. **שילוב Capsule Networks בבעיית ניתוב** — שימוש לא-טריוויאלי בארכיטקטורת רשתות-קפסולות (לרוב משויכת לראיית מכונה) עבור בעיית אופטימיזציה קומבינטורית (VRP).
2. **Dynamic Routing as Soft Clustering** — מנגנון ה"הסכמה" יוצר אשכול רך של תחנות לפי אופי משולב (גיאוגרפיה + זמן + דחיפות), ולא רק לפי קרבה גיאומטרית.
3. **היוריסטיקה הניתנת-לאימון** — מטריצות המשקל קבועות-מראש כעת אך מיושמות כשכבות `nn.Linear` רגילות, כך שבעבודה עתידית ניתן לאמן אותן (Backpropagation) על דאטה אמיתי של מסלולים מוצלחים, בלי לשנות מבנה.
4. **ניהול מצב דינמי וזיהוי כפילויות** — פתרון נקי לבעיית "רעש GPS" ועדכונים אינקרמנטליים, חשוב מאוד למערכת production מול לקוח מובייל.
5. **שקלול רב-גורמי בציון המסלול** — `score = sim − dist×(traffic+priority)` משלב תובנת-מודל (LGCN) עם עלות תפעולית בפועל בנוסחה אחת, פשוטה לכוונון.

---

## 10. מגבלות ועבודה עתידית

- המודל אינו מאומן בפועל (משקלים קבועים) — שלב טבעי הבא הוא אימון על דאטה היסטורי (מסלולים בפועל מול זמני-אספקה).
- בניית המסלול היא Greedy (לא בהכרח אופטימלית גלובלית) — אפשר להרחיב ל-2-opt / Beam Search מעל ציוני ה-`sim`.
- כיום `num_subroutes=1` (נהג בודד); המבנה תומך טבעית בהרחבה למספר נהגים/תת-מסלולים מבלי לשנות שכבות.
- מרחק מחושב כאוקלידי על קואורדינטות (lat/lon) — לא מרחק-כביש בפועל; אינטגרציה עם API ניווט (כגון Google Distance Matrix) תשפר את הדיוק.

---

## 11. סיכום

LGCN הוא היברידיזציה מקורית בין **רשתות קפסולות עצביות** (לכידת "אופי" תחנה דרך וקטור רב-מימדי ודמיון-הסכמה) לבין **היוריסטיקת ניתוב גרידית** קלאסית. הארכיטקטורה מאפשרת לקודד דחיפות, חלון-זמן, מיקום ועדיפות לתוך ציון יחיד (`sim`) באופן הניתן להרחבה ולאימון עתידי, בעוד שכבת ה-API והגרף-הדינמי-לכל-נהג מספקות תשתית פרודקשן מלאה — כולל טיפול ברעש GPS, עדכון אינקרמנטלי, ושמירת מצב.
